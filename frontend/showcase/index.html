What it does

malphas is a peer-to-peer encrypted messenger designed for people who need privacy guarantees stronger than what centralized services can offer. Every message is encrypted, deniably authenticated, and routed through an onion network. Nothing touches disk. Nothing touches a server.

>_

Zero infrastructure

No servers, no cloud, no accounts. Your node runs on your machine. The only external dependency is Tor, and even that is optional.

{}

Onion routing

Messages traverse multiple peers in layered encryption. Each relay sees only the previous and next hop. No relay can read your content or identify the full path.

.onion

Tor hidden services

Your node registers a Tor v3 hidden service derived from your passphrase. No port forwarding, no public IP. Your .onion address is permanent and stable across restarts.

#!

Deniable authentication

Messages use HMAC-SHA256 instead of digital signatures. Both peers can produce the same authentication tag. If a device is seized, no one can cryptographically prove who wrote what.

[=]

Key pinning (TOFU)

On first contact, the peer's Ed25519 key is pinned. If it changes on reconnect, the handshake is rejected and a warning is displayed. Detects impersonation attempts.

/!

Emergency wipe

The /panic command clears all messages, peer data, encryption keys, pin store, and message queues from memory in milliseconds. Then exits. The encrypted address book on disk is noise without the passphrase.

~~

Cover traffic

Encrypted dummy packets sent at random intervals, indistinguishable from real messages on the wire. Makes traffic analysis significantly more expensive for an observer.

@

Signed invites

Run /export to generate a malphas:// URL signed with Ed25519, containing your public keys and .onion address. Your peer runs /import and connects instantly. No manual key exchange.

How it works

1

Launch and derive identity

Enter a passphrase. Argon2id (64MB RAM, ~200ms) derives a 64-byte seed. From that seed: Ed25519 and X25519 keypairs, a .onion address, and an encrypted storage key. Same passphrase always produces the same identity. Nothing is stored.

2

Share your invite

The /export command generates a signed malphas:// URL containing your public keys and .onion address. Send it over any channel — it contains only public information. The privacy comes from the Tor transport, not from secrecy of the invite.

3

Authenticated handshake

Your peer imports your invite and connects through Tor. Both sides sign ephemeral keys with Ed25519, verifying identity. Keys are pinned on first contact. A session key is derived via ECDH + HKDF. An HMAC key is derived for deniable message authentication.

4

Onion-routed messaging

Messages are wrapped in layers of encryption — one per hop. Each relay decrypts its layer, reads the next hop, and forwards. The destination decrypts the final layer. Delivery is confirmed by Ed25519-signed read receipts routed back through the network.

Cryptographic stack

Every primitive comes from the cryptography library (OpenSSL/libssl). No custom cryptography. No novel constructions. Battle-tested algorithms assembled for a specific threat model.

Primitive Algorithm Purpose
Password hashing Argon2id Passphrase to seed (64MB, ~200ms per attempt)
Key derivation HKDF-SHA256 Seed to keys, session to HMAC key
Key exchange X25519 (ECDH) Ephemeral session key establishment
Encryption ChaCha20-Poly1305 All message and storage encryption
Signing Ed25519 Handshake auth, receipts, invites, Tor identity
Message auth HMAC-SHA256 Deniable message authentication
Onion layer X25519 + ChaCha20-Poly1305 Per-hop circuit encryption
