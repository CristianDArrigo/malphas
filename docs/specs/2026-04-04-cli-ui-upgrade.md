# CLI UI Upgrade — Design Spec

## Goal

Replace the current basic CLI with a modern terminal interface using prompt_toolkit (input) and rich (output). Add /export and /import commands for credential sharing.

## Dependencies

Add to pyproject.toml:
- `prompt_toolkit>=3.0`
- `rich>=13.0`

## Layout

Split-pane layout via prompt_toolkit:

```
┌─────────────────────────────────────────────────┐
│                                                 │
│  Message/output area (scrolls)                  │
│  - chat messages                                │
│  - command output (/id, /peers, /book, etc)     │
│  - system notifications                         │
│                                                 │
├─────────────────────────────────────────────────┤
│ [3 peers] [chat: alice] [2 receipts]      [tor] │
├─────────────────────────────────────────────────┤
│ > input prompt (fixed, never moves)             │
└─────────────────────────────────────────────────┘
```

- Top area: all output, auto-scrolls on new content
- Status bar: single line, updates dynamically
- Input: fixed at bottom, supports readline (history, cursor movement)

## Visual Style

Dark base, subtle accents:

| Element | Color |
|---------|-------|
| Sent messages | dim white/grey |
| Received messages | sender label in muted red accent |
| `[ok]` prefix | muted green |
| `[err]` prefix | muted red |
| `[!]` notification | muted yellow |
| Command output boxes | dim border, grey text |
| Status bar background | dark grey (#2a2a2a) |
| Status bar text | dim white, accent values |
| Separators | dim grey lines |
| Prompt character | dim grey `>` |

No emoji. No bold. Austere, dark-side feel.

## Tab Completion

First level: all commands (`/chat`, `/book`, `/add`, `/export`, `/import`, `/id`, `/peers`, `/history`, `/wipe`, `/panic`, `/help`, `/quit`).

Second level (contextual):
- `/chat <tab>` — address book labels + connected peer_ids (first 8 chars)
- `/book rm <tab>` — existing address book labels
- Other commands: no second-level completion

Implementation: prompt_toolkit `Completer` subclass that reads from node.discovery and book at completion time.

## Input History

prompt_toolkit `InMemoryHistory` — arrow up/down cycles through previous commands and messages. Not persisted to disk (consistent with zero-log policy).

## /export Command

Generates a signed invite blob containing the node's public credentials.

Output format:
```
malphas://<base64url of signed JSON>
```

JSON payload (before signing):
```json
{
  "type": "invite",
  "v": 1,
  "peer_id": "a0f8e7d8...",
  "x25519_pub": "3f7a...",
  "ed25519_pub": "8b2c...",
  "host": "192.168.1.10",
  "port": 7777,
  "onion": "abc...xyz.onion"
}
```

- `onion` field present only if node is running with `--tor` and has a registered hidden service
- `host` and `port` are the node's current listen address

Signed blob format:
```
base64url( sig(64 bytes) + json_bytes )
```

Signature: Ed25519 sign the JSON bytes with the node's identity key.

Warning printed after the blob:
```
  this invite contains your current host and port.
  it will stop working if your IP changes.
  if running with --tor, the .onion address is permanent.
```

## /import Command

Syntax: `/import malphas://...`

Steps:
1. Strip `malphas://` prefix
2. base64url decode → sig(64) + json_bytes
3. Parse JSON, extract ed25519_pub
4. Verify Ed25519 signature over json_bytes using ed25519_pub from the payload (self-signed integrity check — proves the invite was generated by someone holding that key)
5. Display summary: peer_id, host:port, .onion (if present)
6. Prompt: "connect? [Y/n]"
7. If yes: connect (prefer .onion if available and --tor active, else host:port)
8. Prompt: "save to address book? [y/N]" → if yes, ask label

## File Changes

### `cli_ui.py` — full rewrite

Replace current implementation with:
- `prompt_toolkit.Application` with `HSplit` layout (output buffer + status bar + input)
- `rich.console.Console` for formatted output (rendered to string, printed into output buffer)
- `MalphasCompleter` class for tab completion
- All existing command methods ported to new output system
- New `/export` and `/import` methods

### `invite.py` — new module

- `generate_invite(identity, host, port, onion=None) -> str` — creates the malphas:// blob
- `parse_invite(blob: str) -> dict` — decodes and verifies, raises ValueError on invalid
- Keep invite logic separate from CLI for testability

### `pyproject.toml`

Add `prompt_toolkit>=3.0` and `rich>=13.0` to dependencies.

### `__main__.py`

No changes — cli_ui.py's `MalphasCLI.run()` interface stays the same.

### Tests

- `test_cli.py` — adapt existing 115 tests to new output format (rich markup in captured output)
- `test_invite.py` — new: generate/parse roundtrip, signature verification, tampered blob rejected, missing fields rejected, v1 format
- New tests for tab completion behavior

## What Does NOT Change

- Command syntax (all `/` commands identical)
- Node, transport, onion routing, crypto, discovery, memory, receipts, obfuscation, addressbook, api
- PWA frontend
- The `MalphasCLI.__init__(node, book)` interface
- The `MalphasCLI.run()` coroutine interface
